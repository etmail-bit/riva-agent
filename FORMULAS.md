# 系統公式總覽

這份文件集中列出 Riva-Agent 飲料店營運效能優化系統裡，網頁與月度報告用到的每一個計算公式，
方便審核與作品集展示。**文件裡的數字都是範本／範例值或官方公開法規費率，不含任何真實店家的
成本結構或薪資數字**——真實參數只存在本機、被 `.gitignore` 排除的 `config/staffing_rules.json`
／`config/cost_rates.json`（範本版 `*.example.json` 有進版控，可以對照欄位名稱）。

每個公式的正式定義都在對應腳本的檔頭 docstring／函式註解裡，這份文件是「總覽＋索引」，
細節異動以程式碼為準；程式碼有異動但這份沒同步更新，以程式碼為準並回頭補這份文件。

---

## 1. 逐時段建議人力（`scripts/calculate_staffing.py`）

```
一般時段：  建議人力 = ceil( 日均杯數 ÷ 單位產能  +  外送耗時小時數 )

開早班時段： 建議人力 = ceil( max(0, 日均杯數 − 開早班前場產能貢獻) ÷ 單位產能
                            + 外送耗時小時數 )
                        + 1（開早班本人的人頭）
```

- **開早班時段**：`config.tea_brewing` 定義的時間窗（起始時間＋時長）。這個時段裡有一位員工
  同時兼顧煮茶跟前場出杯，但他的前場產能被壓低成 `front_capacity_contribution_cups_per_hour`
  （範例值 8 杯/hr），不是完整的 `capacity.cups_per_staff_per_hour`（範例值 30 杯/hr）。
  **只有這一位被打折，同時段如果還缺人，補進來的人一律用完整產能算**；他本人這整段時間
  都算一個班，所以總人力要 `+1`，不是「除了他還要另外配幾個」。
- **外送耗時**：`外送耗時小時數 = 該時段日均外送單數 × fulfillment_minutes_per_order ÷ 60`
  （範例值 25 分鐘/單）。只算「店家自己送」的單（`delivery_count`），平台叫車外送
  （`platform_count`，外部騎士取貨）不佔用店內人力，不計入。
- **函式**：`required_front_staff_for_hour(cups, delivery_hours, is_tea, config)`，回傳
  `{"raw": 取整前數字, "required": 取整後整數, "formula": 帶入實際數字的算式文字}`——網頁跟
  月度報告顯示的「建議人力」都是 `required`，同時會顯示 `formula`（例如 `ceil(2.01)+1`）
  讓使用者看到差多少就會多/少排一個人，不是只看到結論整數。
- 三支腳本共用同一個函式（`calculate_staffing.py`／`estimate_staffing_by_weekday.py`／
  `estimate_staffing_cost.py`），不會各自重複一份、彼此對不上。

## 2. 平日/假日杯量拆分（`scripts/analyze_staffing_daytype.py`）

真實資料反推法（只用在有真實星期六/日單日樣本的月份）：

```
假日均量 = 該月所有真實星期六/日單日樣本的平均值（直接量到的真實數字）

平日均量 = ( 月彙總日均杯數 × 當月天數 − 假日均量 × 當月週末天數 )
           ÷ 當月平日天數
```

兩邊都是真實 POS 資料撐出來的代數關係，不是估計值：月彙總杯數是 POS 系統本來就有的真實數字，
假日均量是使用者額外提供的星期六/日單日時段占比報表真實量到的數字，平日均量是用這兩者反推
出來的（計算出來，但輸入都是真實資料）。

## 3. 星期幾（一~日）建議人力估計（`scripts/estimate_staffing_by_weekday.py`）

給沒有真實星期六/日單日樣本的店用的替代方案，**是推算值，不是實測值**：

```
每張發票約幾杯（轉換比例） = 該月總杯數（真實） ÷ 該月正常發票張數（真實）
                              → 取有真實杯數月份的平均值

估計杯數（某星期幾、某時段） = 該星期幾該時段的日均發票張數（真實交易時間戳記回推）
                                × 每張發票約幾杯

建議人力 = 套用「第 1 節」的建議人力公式，用「估計杯數」代入
```

外送耗時修正沒有星期幾拆分的原始資料，用跨月日均值套用到七天（近似值）。

## 4. 月盈虧（`scripts/calculate_pnl.py`）

```
稅前淨利 = 營收
           − 原物料（cogs）
           − 原物料損耗（cogs × material_waste_pct，範例 4%）
           − 平台抽成（Ubereats/Foodpanda 金額 × 各自費率，範例各 30%）
           − 金流手續費（信用卡/其他電子支付金額 × 各自費率，範例 3%／2%）
           − 人事成本（見第 5 節）
           − 房租 − 水電 − 加盟金攤提
           − 營業稅（應稅營收 × business_tax_pct，範例 5%）

預估所得稅 = max( 帳面稅, 推定稅 )     ← 取較高者，見下方說明

稅後淨利 = 稅前淨利 − 預估所得稅
```

**預估所得稅（2026-07-14 修正，不是單純「稅前淨利 × 稅率」）**：小商家常見用「擴大書審」
簡化申報，稅局規定用兩個算法比較、**取較高者**，不是只看帳面損益：

```
帳面稅 = max(0, 稅前淨利 × corporate_income_tax_pct)     （範例 20%，虧損月份不倒扣）
推定稅 = 營收 × deemed_profit_ratio_pct × corporate_income_tax_pct
         （deemed_profit_ratio_pct 是「同業利潤標準/擴大書審純益率」，由國稅局依行業別
         公告的官方數字，飲料店這個行業別是 6%——這是公開法規資料，不是店家自訂比例）
```

推定稅**不管帳面賺賠都要繳**（用意是防止小商家虛報虧損逃漏稅），所以即使某個月帳面上是
虧損（帳面稅＝0），只要營收不是 0，推定稅通常還是大於 0，很可能變成「取較高者」後實際
要繳的那個數字——這點容易被忽略，之前系統一度只算「帳面稅」，虧損月份所得稅直接算成 0，
低估了實際稅負跟虧損金額，2026-07-14 才修正加回推定稅這條路徑。

`原物料`／`水電`三項若有 `monthly_cost_actuals` 的真實數字會優先使用，沒有才 fallback 用
`config/cost_rates.json` 的概算比例／固定金額（`revenue_source`／`labor_cost_source` 兩個
欄位會分別標記營收跟人事成本這次用的是哪一層來源）。房租、加盟金攤提沒有實際值輸入機制，
一律用 config 固定金額。

## 5. 人事成本三層優先順序（2026-07-14 新增，取代單一概算公式）

```
① 這個店這個月有真實排班資料 → 用第 6 節「真實薪資計算」逐員工加總的公司總負擔成本
   （已經含真實勞健保/職災/勞退，不再乘保費倍數）

② 沒有①，但有手動輸入底薪（monthly_cost_actuals.labor_actual）
   → 人事成本 = labor_actual × (1 + labor_insurance_overhead_pct)（範例 19.6%）

③ 都沒有 → 人事成本 = config 概算底薪 × (1 + labor_insurance_overhead_pct)
```

雲端部署因為刻意不同步逐員工排班明細（零洩漏設計），只會落在②或③，不會出現①。

## 6. 真實薪資計算 / 勞健保精算（`scripts/calculate_payroll.py`）

```
公司總負擔成本（單一員工、單月）
    = 底薪/工資 + 加班費 + 雇主保險負擔

底薪/工資：
    正職（manager/staff）：固定月薪（config.wages）
    兼職（part_time）：當月實際工時 × 時薪

加班費（只有正職才算）：
    = max(0, 每日工時 − daily_regular_hours) 加總 × overtime_hourly（範例值 8 小時／199 元）

雇主保險負擔 = 勞保(雇主70%) + 職災保險(雇主100%) + 健保(雇主60%×1.56眷屬係數) + 勞退(雇主6%)
```

四項保費都是官方公開法規費率（民國 115 年 / 2026 年）：

| 項目 | 費率 | 負擔比例 |
|---|---|---|
| 勞保普通事故 | 11.5% | 員工20%／雇主70%／政府10% |
| 勞保就業保險 | 1.0% | 員工20%／雇主70%／政府10% |
| 職災保險（住宿餐飲業） | 行業費率0.12%＋上下班災害0.07%＝0.19% | 雇主100% |
| 全民健保 | 5.17% | 員工30%／雇主60%（雇主端另乘全國平均眷屬係數1.56） |
| 勞退提繳 | 6% | 雇主100% |

- 投保薪資依「投保薪資分級表」查表對應到一個整數級距金額，不是直接用薪資本身乘費率。
- 勞保/職災保險費率**分開算、各自四捨五入後再相加**，不是合計費率一次算（官方金額表逐點
  核對後發現這樣才會完全對上，見 `calc_labor_insurance()`）。
- 兼職員工月收入低於一般分級表最低級距（29,500）時，勞保/職災保險改查「部分工時低收入」
  特例級距（11,100/12,540 兩級 + 12,540~29,500 銜接段），健保/勞退沒有這條特例。
- 完整分級表、費率、法規來源見 `config/insurance_rates_2026.json`（純公開法規資料，可進
  版控，跟真實薪資數字無關）。

## 7. 排班成本估算（`scripts/estimate_staffing_cost.py`）

```
合理人力（每小時） = max(capacity.min_floor_staff, 第 1 節建議人力公式算出來的人數)
```

- `min_floor_staff`（範例 2 人）是操作面下限：純杯量算出來的建議人力離峰時段可能只有 1 人，
  但現場至少要有人顧收銀＋出杯，這個下限不是杯量算出來的，是操作面硬性要求。
- **保守版**：尖峰時段（`scenario.peak_hours`）用 `peak_floor_staff`（範例 3 人）當緩衝下限。
- **積極版**：全天一律套用 `min_floor_staff`，是純公式算出來的理論上限，只當參考基準。
- 可省下的變動成本 = min(總落差時數, 實際加班+兼職時數)——不會把正職固定底薪時數也算進
  「可省」範圍，那筆錢不會因為調班表就消失。

## 已知的推算值 vs 實測值（審核時要注意的區分）

| 資料 | 等級 |
|---|---|
| 假日單日樣本杯量 | 實測值（使用者額外提供的星期六/日單日報表） |
| 平日杯量（有假日樣本月份） | 反推值（代數關係，兩邊都是真實資料） |
| 星期幾建議人力（`estimate_staffing_by_weekday.py`） | 推算值（發票張數 × 轉換比例，沒有真實假日樣本的店才用） |
| 外送耗時（星期幾/平日假日顆粒度） | 近似值（跨月日均值套用，沒有更細的原始資料） |
| 真實薪資計算（`calculate_payroll.py`） | 實測值（正職）／大概金額（兼職，除非排班原始資料已人工核對過） |
| 月盈虧的營收/人事成本 | 依 `revenue_source`／`labor_cost_source` 欄位標記的那一層而定，網頁會顯示對應提示 |
